## Finding Detail

### C1 — Medium — manifest.json schema is unspecified

**Reviewer:** Architecture (A)

`sidebar.js` fetches `output/<run-N>/manifest.json` and renders the tree.
The `manifest.json` structure is described informally (`{"stages": [...],
"feedback_dir": "feedback/"}`) but no schema is defined. `build_manifest()`
in `tirvi/debug/manifest.py` writes this file. Without a formal schema,
`sidebar.js` and `manifest.py` can drift independently during MVP
development — the JS consumer will silently break when the Python producer
changes field names.

This is acceptable for a POC where both files are in the same repo and
reviewed together, but the risk escalates at MVP when they may be in
separate code paths.

**Requested revision:** Add a `manifest.json` schema sketch (4-6 field
definitions) to `design.md §Interfaces` under the `output/<N>/` filesystem
contract section. Formal JSON Schema document is deferred to MVP.

---

### C2 — Medium — HLD deviation: `output/` directory storage contract

**Reviewer:** HLD Compliance (H)

The HLD deviation table correctly notes that `output/<N>/` is not in the
HLD §5.2 storage model (which uses GCS for production). However the
design does not clarify whether `output/` is tracked in `.gitignore` or
committed. If committed, large binary artifacts (`.png` overlays,
`.mp3` audio) will bloat the repo history. If not committed, the pipeline
run artifacts are ephemeral and the debug viewer cannot be demonstrated
from a clean checkout.

**Requested revision:** Add a line to §Decisions: "D-06: `output/` is
`.gitignore`d — demo artifacts are ephemeral; regenerated by `python
scripts/run_demo.py --debug`."

---

### C3 — Low — `do_POST` in `scripts/run_demo.py` adds executable logic

**Reviewer:** Code Quality / Security (Q)

The design's Risks entry notes "add a tiny `do_POST` for `/feedback` in
`scripts/run_demo.py`." The workflow scaffolding disqualifier rule
(`.claude/rules/workflow.md`) states that adding executable logic to
`scripts/` requires the full design + TDD pipeline. The `do_POST` handler
— even if "tiny" — writes JSON to disk and must handle HTTP parsing,
path traversal prevention (feedback file path must not escape `output/<N>/
feedback/`), and error responses.

This is a Low finding for the deferred MVP design but should be flagged
so the implementer does not treat it as a config-only change.

**Requested revision:** Add a note to §Approach DE-06: "The `/feedback`
POST endpoint in `scripts/run_demo.py` requires a TDD task (path
traversal guard + 400 error on malformed body) and is not scaffolding."

---

### C4 — Low — No test plan; tasks.md contains TBD stubs

**Reviewer:** Test Coverage (T)

`tasks.md` contains placeholder TBD tasks. This is expected for a deferred
MVP feature. The design is well-structured enough that a task breakdown
can be derived at TDD-time: 7 DEs map cleanly to 7+ tasks covering
`three-panel-grid`, `artifact-tree`, `feedback-panel`, `debug-sink`,
`manifest-builder`, `run-numbering`, and `do_POST`.

No revision needed now. Flagged for awareness when TDD scheduling begins.
